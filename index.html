Forge Language

Core Design Specification

Version 0.2-draft Status Design Phase Memory Model Region-Owned

1 Introduction

Forge is a systems programming language with compile-time memory and concurrency safety through a region-based ownership model and RAII-style lifetime tracking. No garbage collector. No explicit lifetime annotation syntax. No borrow-checker gymnastics. Every memory operation has a clear, predictable cost.

Design Philosophy

  • Safety by default. Memory errors, data races, and use-after-free are compile-time errors.
  • Minimal annotation. Lifetimes are inferred. Region parameters are explicit only when control is needed.
  • No hidden costs. No GC pauses, no implicit copies.
  • Beginner approachability. One mental model: regions own values, pointers borrow, moves transfer.

2 Memory Model & Regions

Every value lives in exactly one region. Regions control allocation strategy, lifetime, mutability defaults, and concurrency access.

Core Rule

Regions are linear. They cannot be copied — only moved or borrowed. A value cannot silently exist in two regions simultaneously.

Default Region

The region annotation is optional. When omitted, the region defaults to @stack. This applies to let bindings, parameters, return types, and pointer targets.

let a: i32          = 10;   // i32@stack — default
let b: i32@heap      = 10;   // i32@heap  — explicit
let p: *Point        = &q;   // *Point@stack — default target
let r: *Point@heap   = &h;   // *Point@heap  — explicit

fn add(a: i32, b: i32) -> i32 { ... }
// equivalent to: fn add(a: i32@stack, b: i32@stack) -> i32@stack

3 Pointers & Ownership

Alias Invalidation Rule

Any alias — mutable or immutable — is automatically invalidated the moment its target is moved or destroyed. For cross-region moves, old aliases additionally become type-incompatible (§EC1). Both checks are independent and enforced at compile time.

4 Sequential Aliases & RAII Lifetime Tracking

A mutable alias is live from creation to its last use. After the last use the compiler marks it dead and a new mutable alias to the same value is permitted. At any program point at most one mutable alias to a given value may be live.

let a = &p1;
inc(a);        // last use of `a` — dead after this line
let b = &p1;  // ✓ `a` is dead
inc(b);
print(p1.X);   // ✓ `b` is dead after the inc call above

EC1 Edge Case: Cross-Region Moves

A cross-region move transfers a value's bytes from one region to another and deallocates the source. Because the region is part of the type, the annotated type changes: T@heap moved to a stack variable becomes T@stack.

What Happens to Existing Borrows

Two independent enforcement layers fire simultaneously on any cross-region move:

  1. Standard move invalidation. All aliases to the moved value are invalidated by the move rule (§3).
  2. Type incompatibility. A *T@heap borrow cannot point into a T@stack value. The region mismatch is a type error independently of invalidation.
let h: Point@heap  = Point{ .X=1, .Y=2 };
let a: *Point@heap = &h;
inc(a);

let s: Point@stack = h;   // cross-region move: bytes copied, heap freed
                            // `h` moved; `a` is (1) invalidated AND (2) type-incompatible
inc(a);  // ✗ compile error — dangling alias (rule 1)
         // ✗ compile error — *Point@heap cannot refer to Point@stack (rule 2)

Large Stack Move Warning

Moving a large heap value to stack may overflow the stack frame at runtime. The compiler emits a warning when the moved type exceeds a configurable threshold (default 1 KiB). Suppress with #[allow(large_stack_move)].

EC2 Edge Case: Conditional & Loop Last-Use Tracking

The "last-use" and "join point" descriptions in earlier drafts are intuitive but not implementable as stated. The normative definition is backward dataflow liveness analysis on the control-flow graph (CFG).

Normative Liveness Definition

An alias a is live-in at CFG node N if there exists a path from N to a use of a that does not pass through a kill of a. A kill is a move, destroy, or scope exit. An alias is dead at N if no such path exists. A new mutable alias to the same value is permitted only at program points where the original is dead.

This handles all branch shapes, back edges, break, continue, and early returns uniformly without special cases.

Case 1 — Used in only one branch

let a = &p1;
if cond {
    inc(a);       // use on true-path only
}
// No path from here to a use of `a` — dead.
let b = &p1;   // ✓ allowed

Case 2 — Used after the branch on all paths

let a = &p1;
if cond {
    inc(a);          // use on true-path
} else {
    print("skip");  // no use of `a`
}
inc(a);             // use on all paths — `a` is live throughout the if block
let b = &p1;        // ✓ `a` dead after the final inc

Case 3 — New alias inside branch while original is live post-branch

let a = &p1;
if cond {
    inc(a);
    let b = &p1;  // ✗ `a` live (used after if on all paths) — two mutable aliases coexist
    inc(b);
}
inc(a);

Case 4 — Conditional use inside loop (back-edge keeps alive)

let a = &p1;
loop {
    if flag { inc(a); }
    // Back-edge: path from here → loop top → inc(a). `a` is live.
    let b = &p1;  // ✗ compile error
}

Case 5 — Loop with unconditional break

let a = &p1;
loop {
    inc(a);
    break;    // unconditional — back-edge unreachable
}
// No path from here back to inc(a). `a` is dead.
let b = &p1;  // ✓ allowed

Case 6 — Early return inside branch

fn process(p1: Point) {
    let a = &p1;
    if err {
        log();
        return;   // path ends — `a` not used after this
    }
    inc(a);        // use on surviving path
    let b = &p1;  // ✓ `a` dead after inc
}
Conservative Fallback

When liveness is inconclusive (indirect control flow, function pointers), the compiler treats the alias as live until the end of its enclosing lexical scope. An explicit block scope { let a = &p1; ... } always terminates liveness at the closing brace — the recommended escape hatch for complex cases.

EC3 Edge Case: Shared Region Aliasing & Locking Rules

Rule 1 — Raw mutable borrows into shared are forbidden

A *T@shared mutable borrow is a compile error in all contexts. Mutation must go through Mutex<T>@shared and a lock block.

✓ Correct
let m: Mutex<Point>@shared
    = Mutex::new(Point{...});
lock(m) |g| { g.X += 1; }
✗ Compile Error
let p: Point@shared = Point{...};
let a: *Point@shared = &p; // ✗
a.X += 1;

Rule 2 — Immutable shared values need no lock

A T@shared not wrapped in Mutex is permanently immutable after initialization. Any number of *const T@shared borrows may coexist across threads without synchronization.

let cfg: Config@shared = load_config();   // permanently immutable
spawn { print(cfg.timeout); }            // ✓ no lock needed
spawn { print(cfg.retries); }            // ✓ concurrent reads always safe

Rule 3 — Guard lifetime is strictly the lock block

The guard produced by lock(m) |g| { ... } cannot be stored outside the block, returned from a function, or captured by a spawn. Any escape attempt is a compile error.

let escaped;
lock(m) |g| {
    g.X += 1;       // ✓
    escaped = g;     // ✗ guard cannot escape lock block
}

fn bad() -> *Point {
    lock(m) |g| { return g; }  // ✗ guard lifetime ends at `}`
}

Rule 4 — Guard obeys normal alias rules inside the block

Inside a lock block, g has type *T and is the only live mutable alias to the inner value. Standard exclusivity applies — you cannot create a second mutable alias to g's target.

lock(m) |g| {
    let alias = g;  // ✗ two live mutable aliases to inner value
}
lock(m) |g| {
    inc(g);   // ✓ single alias, last-use tracked normally
    inc(g);   // ✓ previous use was last; new borrow taken here
}

Rule 5 — Nested locks: no compile-time deadlock prevention

The compiler does not statically prevent deadlocks from nested or out-of-order locking. This is a deliberate tradeoff for v1.

Deadlock Risk

Locking two mutexes in different orders on different threads produces a deadlock at runtime. Convention: always acquire multiple locks in a globally consistent order (e.g. by address). A future lock_ordered! macro may enforce this statically.

// Safe — consistent lock order on all threads
lock(m1) |g1| {
    lock(m2) |g2| { g1.X = g2.Y; }  // ✓ if all threads use m1-then-m2
}
// Thread A: lock(m1) then lock(m2)
// Thread B: lock(m2) then lock(m1)  ← deadlock — compiler will NOT catch this

5 Generics & Region Polymorphism

Regions are first-class type parameters. Functions can be generic over both value type T and region R. Region inference applies at call sites.

fn first<T, R>(slice: *[T]@R) -> *T@R { return &slice[0]; }

let items: [i32]@heap = [1, 2, 3];
let x = first(items);  // R inferred as heap

6 Constraints

constraint is a keyword. Kind — region or type — is inferred from the RHS. Mixing region names and type/interface names on one RHS is a compile error. Applying a region constraint to a type parameter or vice versa is a compile error caught at the function definition.

constraint Allocable  = heap | stack;            // region constraint
constraint Mutable    = heap | stack | tls;    // region constraint
constraint ThreadSafe = shared | static;        // region constraint
constraint Numeric    = i32 | i64 | f32 | f64; // type constraint
constraint Printable  = Stringer;                // type constraint (interface)

fn clone<T, R: Allocable>(p: *T@R) -> T@R { ... }  // ✓ region constraint on region param
fn sum<T: Numeric>(a: T, b: T) -> T           { ... }  // ✓ type constraint on type param
fn bad1<T: Allocable>(...) { ... }  // ✗ region constraint on type param
fn bad2<R: Numeric>(...)   { ... }  // ✗ type constraint on region param

7 Methods

Methods use Go-style receiver syntax, declared outside the type definition. The compiler auto-takes a reference when calling a pointer-receiver method on a non-pointer value.

struct Point { X: f32, Y: f32 }

fn (Point) length(self) -> f32 {            // value receiver — copy
    return sqrt(self.X*self.X + self.Y*self.Y);
}
fn (*Point) scale(self, f: f32) {            // pointer receiver — mutates in place
    self.X *= f; self.Y *= f;
}
fn (*Point@R) translate<R: Mutable>(self, dx: f32, dy: f32) {
    self.X += dx; self.Y += dy;   // works on heap, stack, tls — not static
}

let p: Point@heap = Point{ .X=3.0, .Y=4.0 };
p.scale(2.0);   // compiler auto-takes &p — no manual & needed

7.1 Interfaces

Any type implementing all methods of an interface implicitly satisfies it. No implements keyword. Structural typing identical to Go. Interface values use dynamic dispatch (vtable); generic constraints use static dispatch (monomorphization).

interface Shape {
    fn area(self) -> f32;
    fn perimeter(self) -> f32;
}
struct Circle { radius: f32 }
fn (Circle) area(self) -> f32      { return 3.14159 * self.radius * self.radius; }
fn (Circle) perimeter(self) -> f32 { return 2.0 * 3.14159 * self.radius; }
// Circle implicitly satisfies Shape

fn largest<T: Shape>(items: [T]) -> f32 { ... }  // static dispatch
fn print_area(s: Shape)                  { ... }  // dynamic dispatch

7.2 Receiver Regions & Interface Satisfaction

An interface method may specify a receiver region. If specified, only that region satisfies it. If omitted, any region satisfies it.

interface HeapMutator { fn reset(*self@heap); }  // heap-only
interface AnyMutator  { fn reset(*self);      }  // any region satisfies

8 Formal Lifetime Rules

Alias liveness is determined by backward dataflow analysis on the CFG (full treatment in §EC2). An alias is live at a program point if a use can be reached from that point without passing through a kill. Dead otherwise.

8.1 Conditionals

✓ One-branch use — dead after block
let a = &p1;
if cond { inc(a); }
// no future use — a dead
let b = &p1;  // ✓
✗ New alias while live post-branch
let a = &p1;
if cond {
    inc(a);
    let b = &p1; // ✗ a live (used after if)
}
inc(a);

8.2 Loops

✓ Unconditional break — back-edge dead
let a = &p1;
loop { inc(a); break; }
let b = &p1;  // ✓
✗ Back-edge keeps alias live
let a = &p1;
loop {
    if f { inc(a); }
    let b = &p1; // ✗ back-edge reaches inc(a)
}

8.3 Conditional Move Semantics

A value moved on any branch is marked possibly-moved after the join. Using it is a compile error unless resolved by one of three patterns. Forge never copies silently.

// 1. Move on all branches — definitively moved
if c { consume(p1); } else { consume(p1); }
// p1 definitively dead — any use is a compile error

// 2. Reinitialize on the moving branch
if c { consume(p1); p1 = Point{ .X=0, .Y=0 }; }
inc(&p1);  // ✓ valid on both paths

// 3. is_valid guard — runtime check for complex flow
if c { consume(p1); }
if is_valid(p1) { inc(&p1); }  // ✓

9 Shared Regions & Concurrency

The shared region is cross-thread accessible. Immutable access needs no synchronization. Mutable access requires Mutex<T>. Raw mutable borrows into shared are forbidden (§EC3 Rule 1). Guard lifetime is strictly the lock block (§EC3 Rule 3).

9.1 Value-Level Locking

let m: Mutex<Point>@shared = Mutex::new(Point{ .X=0, .Y=0 });
lock(m) |g| {
    g.X += 1;   // g: *Point — exclusive mutable access
}               // mutex released; g lifetime ends

9.2 Parallel Safety

✓ Independent values
spawn { inc(&p1); }
spawn { inc(&p2); }  // ✓
✗ Same value — data race
spawn { inc(&p1); }
spawn { inc(&p1); }  // ✗

10 Destruction & Drop Hooks

Values are destroyed explicitly via destroy or implicitly on scope exit. A custom drop hook runs before deallocation. For user-defined arena regions, hooks fire in reverse allocation order before backing memory is freed.

impl Drop for FileHandle {
    fn drop(self: *FileHandle) { close_fd(self.fd); }
}
{
    let f: FileHandle = open("data.bin");
}  // drop hook fires here — fd closed before stack frame is popped

11 Unsafe Blocks

unsafe { ... } opts out of alias and move checking within the block. Values crossing the unsafe/safe boundary must satisfy all normal alias and region rules at the crossing point.

unsafe {
    let raw: *u8@heap = raw_alloc(64);
    raw_free(raw);
}  // `raw` cannot be used outside — all rules apply at the boundary
