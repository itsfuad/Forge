Forge Language

Core Design Specification

Version 0.1-draft Status Design Phase Memory Model Region-Owned

1 Introduction

Forge is a systems programming language designed for low-level control with compile-time memory and concurrency safety. It achieves this through a region-based ownership model with RAII-style lifetime tracking, eliminating the need for a garbage collector, explicit lifetime annotations, or verbose borrow checker gymnastics.

Forge targets developers who want Rust-level safety guarantees with a significantly lower annotation burden.

Design Philosophy

  • Safety by default. Memory errors, data races, and use-after-free are compile-time errors, not runtime surprises.
  • Minimal annotation. Lifetimes are inferred. Region parameters are explicit only when the programmer needs control.
  • No hidden costs. Every memory operation has a clear, predictable meaning. No GC pauses, no implicit copies.
  • Beginner approachability. The mental model — regions own values, pointers borrow them, moves transfer them — is consistent and learnable.

2 Memory Model & Regions

Regions are the fundamental unit of memory management in Forge. Every value lives in exactly one region. Regions control allocation strategy, lifetime, mutability defaults, and concurrency access.

Core Rule

Regions are linear. They cannot be copied; they can only be moved or borrowed via pointers. A value cannot silently exist in two regions at once.

Built-in Region Types

Declaring Values in Regions

let p1: Point@heap   = Point{ .X=0, .Y=0 };
let p2: Point@stack  = Point{ .X=1, .Y=1 };
let p3: Point@static = Point{ .X=0, .Y=0 }; // immutable by default

User-Defined Regions

Programmers can declare named regions for arena-style allocation or custom storage lifetimes. A user-defined region can have a custom drop hook (see §8), which runs when the region goes out of scope, freeing all contained memory in one operation.

region scratch: heap;          // heap-backed named region
let buf: Buffer@scratch = ...;  // lives in the scratch arena
// when `scratch` exits scope, all contents are freed at once

3 Pointers & Ownership

SyntaxMeaningOwnership Transfer
T@ROwned value in region RYes — moves ownership, invalidates source
*T@RMutable borrow (pointer) into region RNo — borrows, compiler tracks last-use
*const T@RImmutable borrow into region RNo — read-only, multiple allowed concurrently
Alias Invalidation Rule

Any alias (*T@R or *const T@R) to a value is automatically invalidated the moment that value is moved or destroyed. Using an invalidated alias is a compile-time error.

Borrow vs. Move Example

fn inc(p: *Point@heap) {
    p.X += 1;
}

fn consume(p: Point@heap) {  // takes ownership
    destroy p;
}

let p1: Point@heap = Point{ .X=0, .Y=0 };
let a = &p1;
inc(a);       // ✓ safe borrow
consume(p1); // ✓ ownership moved — `a` is now invalidated by compiler
inc(a);       // ✗ compile error: `a` is a dangling alias

4 Sequential Aliases & RAII-Style Lifetime Tracking

Forge uses last-use tracking to allow sequential mutable aliases without explicit scoping. A mutable alias is considered live from its point of creation to its last use. After the last use, the compiler treats it as dead, and a new mutable alias to the same value is permitted.

Exclusivity Rule

At any given program point, at most one mutable alias (*T@R) to a given value may be live. Multiple immutable aliases (*const T@R) may coexist, but not simultaneously with a mutable alias.

let a = &p1;
inc(a);        // last use of `a` — compiler marks it dead after this line
let b = &p1;  // ✓ allowed: `a` is dead
inc(b);
print(p1.X);  // ✓ allowed: `b` is dead after the inc call above
Escape Hatch: Explicit Scope

When the compiler's last-use inference is insufficient (e.g. inside conditional branches), use a block scope to explicitly bound an alias's lifetime. The alias provably dies at the closing brace.

{
    let a = &p1;
    inc(a);
}  // `a` is provably dead here — no ambiguity
let b = &p1;  // ✓ unambiguously allowed

5 Default Region

The region annotation on a type is optional. When omitted, the region defaults to @stack. This keeps common local variable declarations clean while still permitting explicit region control whenever needed.

let a: i32 = 10;            // i32@stack  — default
let b: i32@heap = 10;        // i32@heap   — explicit
let c: Point = Point{...};   // Point@stack — default
let d: Point@heap = Point{...}; // Point@heap  — explicit

The same rule applies to function parameters and return types:

fn add(a: i32, b: i32) -> i32 { ... }
// equivalent to: fn add(a: i32@stack, b: i32@stack) -> i32@stack

fn make_point() -> Point@heap { ... }
// explicit — caller knows the result is heap-allocated
Pointers & Default Region

When a pointer type omits the region, it defaults to pointing into @stack: *Point means *Point@stack. If you take a reference to a heap value, the region must be stated explicitly: let p: *Point@heap = &heap_val;

6 Generics & Region Polymorphism

Regions are first-class type parameters. A function can be generic over both a value type T and a region R. Region inference applies at call sites, so callers rarely need to name the region explicitly.

// Explicit region parameter — R inferred at call site
fn first<T, R>(slice: *[T]@R) -> *T@R {
    return &slice[0];
}

let items: [i32]@heap = [1, 2, 3];
let x = first(items);  // R inferred as `heap` — caller writes nothing

7 Constraints

constraint is a keyword for defining named constraint groups. Constraints come in two kinds — region constraints and type constraints — and the kind is inferred from the right-hand side at declaration time.

Constraint Separation Rule

A region constraint may only be applied to a region parameter. A type constraint may only be applied to a type parameter. Cross-applying one kind to the other is a compile-time error. Mixing region names and type names on the right-hand side of a constraint declaration is also a compile-time error.

Declaring Constraints

// Region constraint — RHS contains only region names
constraint Allocable = heap | stack;
constraint Mutable   = heap | stack | tls;  // excludes `static`
constraint ThreadSafe = shared | static;

// Type constraint — RHS contains only types or interfaces
constraint Numeric   = i32 | i64 | f32 | f64;
constraint Printable = Stringer;  // any type satisfying the Stringer interface

// ✗ Compile error — cannot mix region names and type names in one constraint
constraint Bad = heap | i32; // ✗

Using Constraints

// R is a region parameter — Allocable (region constraint) applies ✓
fn clone<T, R: Allocable>(p: *T@R) -> T@R {
    let copy: T@R = allocate_in<R>();
    *copy = *p;
    return copy;
}

// T is a type parameter — Numeric (type constraint) applies ✓
fn sum<T: Numeric>(a: T, b: T) -> T {
    return a + b;
}

// ✗ Compile error — Allocable is a region constraint, T is a type parameter
fn bad<T: Allocable>(...) // ✗

// ✗ Compile error — Numeric is a type constraint, R is a region parameter
fn bad<R: Numeric>(...) // ✗

Inline region/type names are also valid without a named constraint:

fn inc<R: heap | stack>(p: *Point@R) { p.X += 1; }
fn double<T: i32 | i64>(x: T) -> T   { return x * 2; }
Constraint Violation = Compile Error at Definition

Constraint violations are caught when the function body is type-checked, not only at call sites. A function that mutates through a region-polymorphic pointer without a Mutable-equivalent constraint will fail at its own definition.

8 Methods

Forge uses a Go-style receiver syntax for methods. Methods are declared outside the type definition, attached via a receiver parameter. The receiver is a pointer or value of the type, and its region annotation follows the same rules as any other type annotation (default @stack).

struct Point { X: f32, Y: f32 }

// Value receiver — gets a copy; cannot mutate original
fn (Point) length(self) -> f32 {
    return sqrt(self.X*self.X + self.Y*self.Y);
}

// Pointer receiver — mutates in place
fn (*Point) scale(self, factor: f32) {
    self.X *= factor;
    self.Y *= factor;
}

// Explicit region on receiver
fn (*Point@heap) move_to_origin(self) {
    self.X = 0.0;
    self.Y = 0.0;
}

Calling Methods

let p: Point@heap = Point{ .X=3.0, .Y=4.0 };

let len = p.length();    // value receiver — copy made
p.scale(2.0);            // pointer receiver — p mutated in place
                          // compiler auto-takes &p when receiver is *T
Auto-Reference Rule

When calling a pointer-receiver method on a non-pointer value, the compiler automatically takes a reference. The caller never needs to write &p.scale(...). This is consistent with Go's behavior.

Region-Polymorphic Receivers

A method can be generic over the receiver's region:

fn (*Point@R) translate<R: Mutable>(self, dx: f32, dy: f32) {
    self.X += dx;
    self.Y += dy;
}
// Works on Point@heap, Point@stack, Point@tls — but not Point@static

8.1 Interfaces

Interfaces are declared as named sets of method signatures. Any type that implements all methods of an interface implicitly satisfies it — no implements keyword or explicit registration required. This is identical to Go's structural typing.

interface Shape {
    fn area(self) -> f32;
    fn perimeter(self) -> f32;
}

interface Stringer {
    fn to_string(self) -> str;
}

// Point satisfies Shape if both methods are defined on it
struct Circle { radius: f32 }

fn (Circle) area(self) -> f32      { return 3.14159 * self.radius * self.radius; }
fn (Circle) perimeter(self) -> f32 { return 2.0 * 3.14159 * self.radius; }
// Circle now implicitly satisfies Shape — no declaration needed

Using Interfaces as Types

fn print_area(s: Shape) {
    print(s.area());
}

let c = Circle{ .radius=5.0 };
print_area(c);   // ✓ Circle satisfies Shape

Interfaces as Type Constraints

Interfaces can be used directly as type constraints, giving compile-time (monomorphized) dispatch rather than dynamic dispatch:

// Static dispatch — T must satisfy Shape at compile time
fn largest_area<T: Shape>(items: [T]) -> f32 {
    let max: f32 = 0.0;
    for item in items { if item.area() > max { max = item.area(); } }
    return max;
}

// Dynamic dispatch — interface value, resolved at runtime
fn print_any_area(s: Shape) { print(s.area()); }

8.2 Receiver Regions & Interface Satisfaction

An interface method signature may include a region annotation on the receiver. If it does, only types whose receiver method matches that region satisfy the interface. If the interface omits the region, any region satisfies it.

interface HeapMutator {
    fn reset(*self@heap);   // receiver must be heap-pointer specifically
}

interface AnyMutator {
    fn reset(*self);         // any region satisfies this
}

10 Formal Lifetime Rules

Join-Point Lifetime Rule

An alias a is live from its point of creation to its join point — the earliest program point that all control-flow paths from a's creation converge on, after which a is no longer used on any path. At and after the join point, a is dead and a new alias to the same value is permitted.

10.1 Conditionals

The join point of an if/else is the first instruction after the entire if block. If an alias is used on any branch, it remains live until after the full block exits.

✓ Allowed
let a = &p1;
if cond {
    inc(a);   // use on one path
}
inc(a);       // use on both paths
              // join point — a dies here
let b = &p1; // ✓ safe
✗ Compile Error
let a = &p1;
if cond {
    inc(a);
    // a still live at this join point
    let b = &p1; // ✗ two mutable aliases live
    inc(b);
}

When both branches use the alias, the join point is after the full if/else:

let a = &p1;
if cond {
    inc(a);          // use on true-path
} else {
    double_inc(a);   // use on false-path
}
// join point — a is dead here (used on all paths, both now ended)
let b = &p1;         // ✓ allowed

When only one branch uses the alias, it is still live until after the full block:

let a = &p1;
if cond {
    inc(a);           // use on true-path only
} else {
    print("no-op");   // no use of `a`
}
// join point — a dead here (conservative: if `a` might be used on any path,
// it is live until all paths have exited)
let b = &p1;          // ✓ allowed

10.2 Loops

If an alias is used anywhere inside a loop body, it remains live for the entire loop duration. A new alias to the same value cannot be created inside the loop body if the original is still live.

✓ Allowed
let a = &p1;
loop {
    inc(a);  // a live entire loop
}
// a dead after loop exits
let b = &p1; // ✓
✗ Compile Error
let a = &p1;
loop {
    inc(a);
    let b = &p1; // ✗ a still live
    inc(b);
}

11 Conditional Move Semantics

A conditional move occurs when ownership of a value is transferred on only one control-flow path. After the branch, the compiler cannot statically guarantee whether the value is still valid.

Conditional Move Rule

If a value is moved on any branch of a conditional, the compiler marks it as possibly moved after the join point. Using a possibly-moved value is a compile-time error unless the programmer explicitly handles both cases.

Handling Conditional Moves

There are three valid patterns:

1. Move on all branches — value is definitively moved, no use permitted after:

if cond {
    consume(p1);   // moved on true-path
} else {
    consume(p1);   // moved on false-path
}
// p1 is definitively moved — any use here is a compile error

2. Reinitialize after conditional move:

if cond {
    consume(p1);                         // moved on true-path
    p1 = Point{ .X=0, .Y=0 };           // reinitialized on true-path
}
inc(&p1); // ✓ safe — p1 is valid on both paths

3. Explicitly guard with is_valid (for complex control flow):

if cond { consume(p1); }

if is_valid(p1) {    // compiler-known dynamic check for possibly-moved values
    inc(&p1);
}
No Implicit Copies on Move

Forge never silently copies a value to resolve a conditional move ambiguity. If the programmer does not reinitialize or guard, the compiler emits an error and requires an explicit resolution.

12 Shared Regions & Concurrency

The shared region is accessible across threads. Immutable access to shared values is always safe and requires no synchronization. Mutable access requires a lock.

8.1 Value-Level Locking

Locking Granularity Rule

Locks in Forge are attached to values, not regions. Locking a whole region would be too coarse for real programs and would contradict Forge's value-level exclusivity model.

Mutable values in the shared region must be wrapped in Mutex<T>:

let p: Mutex<Point>@shared = Mutex::new(Point{ .X=0, .Y=0 });

// On thread A:
lock(p) |guard| {
    guard.X += 1;  // exclusive mutable access inside lock block
}

// On thread B (simultaneous — safe, will block until A releases):
lock(p) |guard| {
    guard.Y += 1;
}

Immutable access to shared values does not require a lock:

let config: Config@shared = load_config();  // immutable after init
// Any thread may read `config` freely — no lock needed
print(config.max_connections);

8.2 Parallel Safety Guarantees

Forge enforces value-level exclusivity at compile time. Two spawn blocks cannot hold simultaneous mutable references to the same value.

✓ Safe — different values
spawn { inc(&p1); }
spawn { inc(&p2); }
// p1 and p2 are independent
✗ Compile Error — same value
spawn { inc(&p1); }
spawn { inc(&p1); }
// ✗ data race: two mutable aliases
//   to p1 across threads

Shared immutable access across threads is always permitted:

let data: [i32]@static = [1, 2, 3];
spawn { print(data[0]); }   // ✓ immutable read
spawn { print(data[1]); }   // ✓ immutable read — concurrent fine

13 Destruction & RAII Hooks

Values are destroyed either explicitly via destroy, or implicitly when their owning region exits scope. Programmers can define a custom drop hook that runs before memory is freed.

// Explicit destruction
let p: Point@heap = ...;
destroy p;  // drop hook runs, then memory freed

// Implicit destruction (stack)
{
    let q: Point@stack = ...;
}  // q's drop hook runs here automatically

// User-defined drop hook
impl Drop for FileHandle {
    fn drop(self: *FileHandle) {
        close_fd(self.fd);   // OS handle released before memory freed
    }
}
Arena Drop

When a user-defined region exits scope, all drop hooks for values within it are called in reverse allocation order before the region's backing memory is freed. This gives arena-style performance with RAII-style correctness.

14 Unsafe Blocks

The unsafe { ... } block explicitly opts out of Forge's alias and move checking within that block. It is intended for FFI, hand-written allocators, and other cases where the programmer can prove safety by means outside the type system.

unsafe {
    let raw: *u8@heap = raw_alloc(64);
    // No alias or lifetime tracking within this block
    // Programmer is responsible for correctness
    raw_free(raw);
}
Unsafe Boundary Rule

An unsafe block may not expose raw pointers or untracked aliases to safe code. Any value crossing the unsafe/safe boundary must satisfy all normal alias and region rules at the point of crossing.
